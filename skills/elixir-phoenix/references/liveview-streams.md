# LiveView streams

Streams are LiveView's mechanism for rendering collections without holding
every item in socket state. The LiveView process keeps a list of *ids*
plus a small bit of bookkeeping; items live on the client only. This is
the difference between a chat room with 10k messages crashing the
WebSocket process (assigning a list to `:messages` is a slow-motion OOM)
and one that scrolls smoothly.

Use streams for any collection that grows, gets filtered, or has items
added/removed over time. If you're tempted to write
`assign(socket, :messages, messages)` and you don't know the upper bound
on length, use a stream instead.

## Setup

In `mount/3`:

    socket = stream(socket, :messages, list_messages())
    # or to start empty:
    socket = stream(socket, :messages, [])

In `handle_event` / `handle_info`, append:

    stream(socket, :messages, [new_msg])

Prepend (at the top of the list, useful for newest-first feeds):

    stream(socket, :messages, [new_msg], at: -1)

Note: `at: -1` prepends and `at: 0` (the default) appends. The semantics
are about DOM insertion position relative to existing items, not list
indexing.

## Template structure

The container needs `phx-update="stream"` and a DOM id; each child must
use the id LiveView assigns:

    <div id="messages" phx-update="stream">
      <div :for={{id, msg} <- @streams.messages} id={id}>
        {msg.text}
      </div>
    </div>

The `id` in `{id, msg}` is a stable DOM id generated by the stream (it
looks like `messages-42`). Use it on the child element so LiveView can
target updates and deletes.

## Filtering, sorting, or otherwise re-deriving the list

Streams aren't enumerable. You can't `Enum.filter` or `Enum.reject` over
them — the LiveView process doesn't have the items. To filter or refresh
the visible set, re-fetch and re-stream with `reset: true`:

    def handle_event("filter", %{"filter" => filter}, socket) do
      messages = list_messages(filter)

      {:noreply,
       socket
       |> assign(:messages_empty?, messages == [])
       |> stream(:messages, messages, reset: true)}
    end

`reset: true` tells LiveView to clear the existing stream and replace it
with the new list. Without it, the new items get *appended* and you end
up with duplicates.

## Deleting an item

    stream_delete(socket, :messages, msg)

The item identity is determined by `msg.id` (or whatever the stream's
`:dom_id` function returns).

## Mutating an item that should re-render

If you update an assign (`@editing_message_id`) that controls how an
existing streamed item should render, you must re-insert that item into
the stream. The render only re-runs for items that have been pushed to
the stream — it doesn't re-render the whole list just because some other
assign changed.

    def handle_event("edit_message", %{"message_id" => id}, socket) do
      message = Chat.get_message!(id)
      edit_form = to_form(Chat.change_message(message))

      {:noreply,
       socket
       |> stream_insert(:messages, message)
       |> assign(:editing_message_id, String.to_integer(id))
       |> assign(:edit_form, edit_form)}
    end

In the template, the `@editing_message_id` is consulted inside the
streamed `<div>`, which now re-renders because the item was re-inserted:

    <div id="messages" phx-update="stream">
      <div :for={{id, message} <- @streams.messages} id={id}>
        {message.username}
        <%= if @editing_message_id == message.id do %>
          <.form for={@edit_form} id={"edit-form-#{message.id}"} phx-submit="save_edit">
            ...
          </.form>
        <% end %>
      </div>
    </div>

## Counts and empty states

Streams don't expose a count to the LiveView. If you need to show a
count, track it in a separate assign and update it whenever the stream
changes.

For empty states, the cleanest approach is the Tailwind `only:` variant,
which targets an element when it's the only child of its parent:

    <div id="tasks" phx-update="stream">
      <div class="hidden only:block">No tasks yet</div>
      <div :for={{id, task} <- @streams.tasks} id={id}>
        {task.name}
      </div>
    </div>

When the stream has items, the empty-state div is hidden (it's no longer
the only child). When the stream is empty, it shows. This pattern only
works when the empty-state element is the sole sibling of the stream
comprehension — anything else in the container breaks the `only:` check.

## Deprecated: `phx-update="append"` and `phx-update="prepend"`

These predate streams. Don't use them — they don't get the memory or DOM
benefits, and they're being phased out.
